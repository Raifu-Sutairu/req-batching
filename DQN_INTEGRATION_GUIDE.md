## 1. Why DQN? The Theory First

Before writing any code it helps to understand *why* DQN is a meaningful comparison and *where* it will structurally differ from PPO.

### PPO (What you have now)
PPO is an **on-policy actor-critic** algorithm. It maintains two networks:
- **Actor (Policy network):** learns a *probability distribution* over actions.
- **Critic (Value network):** learns to estimate how good the current state is.

It updates both networks using experience collected *right now* (on-policy), throws that experience away, and repeats.

**Key trait:** Very stable training, good at continuous state spaces like yours.

### DQN (What you're adding)
DQN is an **off-policy value-based** algorithm. It maintains one network:
- **Q-network:** learns `Q(s, a)` — the expected total future reward of taking action `a` in state `s`.

It does **not** maintain a policy network. Instead it derives the policy by choosing the action with the highest Q-value:
```
action = argmax_a  Q(state, action)
```

It stores all past experience in a **Replay Buffer** and samples randomly from it during training — which is why it's off-policy.

**Key traits:**
- More sample-efficient (reuses old data).
- Prone to instability without tricks like Target Networks and Experience Replay.
- Does **not** natively handle exploration on its own — uses an **ε-greedy** strategy (take a random action with probability ε, greedy action otherwise).

### Why This Comparison is Academically Interesting For Your Project

| Property | PPO | DQN |
|---|---|---|
| Policy type | Stochastic (probabilistic) | Deterministic (argmax) |
| Exploration | Entropy bonus (`ent_coef`) | ε-greedy decay |
| Data usage | On-policy (1 use per sample) | Off-policy (replay buffer, reused many times) |
| Stability | Very stable | Can diverge without careful tuning |
| Best for | Continuous or large action spaces | Small discrete action spaces |
| Parallelism | Multiple envs (`n_envs=4`) | Typically single env |

Your environment has `Discrete(2)` actions — DQN is theoretically well-suited to this. But PPO does better with the continuous 6-dim state. This tension is exactly what makes the comparison interesting.

## 3. Step 1 — Add DQN Hyperparameters to `config.py`

Open `config.py` and add the following block **after** the existing `EXPERIMENT_CONFIGS` dict:

```python
# ───────────────────────────────────────────────────────────────────────────
# DQN Hyperparameters
# Compare directly against PPO_KWARGS in agent/train.py
# ───────────────────────────────────────────────────────────────────────────

DQN_CONFIG = {
    # Learning
    "learning_rate": 1e-4,         # DQN is more sensitive to LR than PPO
    "gamma": 0.99,                 # Discount factor — same as PPO for fair comparison
    "batch_size": 64,              # Minibatch size for Q-network updates — same as PPO

    # Replay buffer
    "buffer_size": 100_000,        # How many past (s,a,r,s') transitions to store
                                   # Key DQN innovation — breaks temporal correlations
    "learning_starts": 10_000,     # Steps of random action BEFORE learning begins
                                   # Fills the replay buffer with diverse data first

    # Target network
    "target_update_interval": 1000, # Every N steps, copy Q-network → Target Q-network
                                    # Prevents the Q-value target from "chasing itself"
                                    # (main stability trick of DQN vs vanilla Q-learning)
    "tau": 1.0,                    # 1.0 = hard update (copy weights completely)
                                   # <1.0 = soft/polyak update (blend weights gradually)

    # Exploration (ε-greedy)
    "exploration_fraction": 0.2,   # Fraction of training spent decaying ε
    "exploration_initial_eps": 1.0, # Start fully random (100% explore)
    "exploration_final_eps": 0.05,  # End at 5% random exploration

    # Network
    "net_arch": [64, 64],          # Same hidden layer size as PPO for fair comparison

    # Training
    "train_freq": 4,               # Train the Q-network every N environment steps
    "gradient_steps": 1,           # Gradient update steps per training call

    # Logging
    "verbose": 1,
    "tensorboard_log": "tensorboard_logs/",
}
```

**Why these values?**
- `buffer_size=100_000`: Large enough to hold roughly 1.5 minutes of simulated experience at 1100 steps/sec. A too-small buffer causes catastrophic forgetting.
- `learning_starts=10_000`: DQN should not start learning on an empty/near-empty buffer — the first 10k steps fill it with random (diverse) experience.
- `target_update_interval=1000`: The Target Network is a *frozen copy* of the Q-network used when computing Q-value targets. Without it, the Q-value target and the estimated Q-value would both update simultaneously, causing the network to "chase a moving target" and diverge.
- `exploration_fraction=0.2`: ε decays from 1.0 → 0.05 over the first 20% of training (first 100k steps if training for 500k total). This is the DQN equivalent of PPO's `ent_coef`.

## 4. Step 2 — Create `agent/train_dqn.py`

Create a new file `agent/train_dqn.py` with the following content:

```python
"""
agent/train_dqn.py

DQN training script for the Dynamic Request Batching agent.

Uses Stable-Baselines3 DQN with:
  - Single env (DQN does not natively support vectorised envs in SB3)
  - MlpPolicy with [64, 64] hidden layers
  - Experience replay buffer of 100k transitions
  - Hard target network updates every 1000 steps
  - ε-greedy exploration decaying 1.0 → 0.05 over first 20% of training
  - EvalCallback → saves best model to models/dqn_best/
  - CheckpointCallback → saves every 50k steps
  - TensorBoard logging → tensorboard_logs/DQN_batching/

Key difference from train.py (PPO):
  DQN uses a single environment and a replay buffer, while PPO uses 4
  parallel environments and discards experience after each update.

Usage:
    python agent/train_dqn.py

View training in TensorBoard:
    tensorboard --logdir tensorboard_logs/
"""

import os
import sys
import time

ROOT = os.path.abspath(os.path.join(os.path.dirname(__file__), ".."))
sys.path.insert(0, ROOT)

from stable_baselines3 import DQN
from stable_baselines3.common.callbacks import (
    EvalCallback,
    CheckpointCallback,
    CallbackList,
)
from stable_baselines3.common.monitor import Monitor

from env.batching_env import BatchingEnv
from config import CONFIG, DQN_CONFIG

# ── Directories ──────────────────────────────────────────────────────────────
TENSORBOARD_DIR       = os.path.join(ROOT, "tensorboard_logs")
MODELS_DIR            = os.path.join(ROOT, "models")
DQN_BEST_MODEL_DIR    = os.path.join(MODELS_DIR, "dqn_best")
DQN_CHECKPOINT_DIR    = os.path.join(MODELS_DIR, "dqn_checkpoints")
MONITOR_LOG_DIR       = os.path.join(ROOT, "monitor_logs")
DQN_FINAL_MODEL_PATH  = os.path.join(MODELS_DIR, "dqn_batching_final")

for d in [DQN_BEST_MODEL_DIR, DQN_CHECKPOINT_DIR, MONITOR_LOG_DIR]:
    os.makedirs(d, exist_ok=True)

TOTAL_TIMESTEPS = 500_000
EVAL_FREQ       = 10_000
CHECKPOINT_FREQ = 50_000


def train():
    print("=" * 60)
    print("  Dynamic Request Batching — DQN Training")
    print("=" * 60)
    print(f"  Total timesteps     : {TOTAL_TIMESTEPS:,}")
    print(f"  Replay buffer size  : {DQN_CONFIG['buffer_size']:,}")
    print(f"  Learning starts at  : {DQN_CONFIG['learning_starts']:,} steps")
    print(f"  Target update every : {DQN_CONFIG['target_update_interval']} steps")
    print(f"  Exploration: ε 1.0 → {DQN_CONFIG['exploration_final_eps']}")
    print(f"  TensorBoard         : {TENSORBOARD_DIR}")
    print("=" * 60)

    # ── Training env (single env — DQN doesn't need vectorised) ─────────────
    # DQN uses a replay buffer, so it does NOT require parallel envs to gather
    # diverse data. The replay buffer itself provides diversity across time.
    train_env = Monitor(
        BatchingEnv(seed=0),
        filename=os.path.join(MONITOR_LOG_DIR, "dqn_train")
    )

    # ── Eval env ──────────────────────────────────────────────────────────────
    eval_env = Monitor(BatchingEnv(seed=9999))

    # ── DQN Model ─────────────────────────────────────────────────────────────
    # policy_kwargs specifies the neural network architecture for the Q-network.
    # We use the same [64, 64] as PPO so the comparison is fair.
    model = DQN(
        "MlpPolicy",
        train_env,
        learning_rate=DQN_CONFIG["learning_rate"],
        buffer_size=DQN_CONFIG["buffer_size"],
        learning_starts=DQN_CONFIG["learning_starts"],
        batch_size=DQN_CONFIG["batch_size"],
        gamma=DQN_CONFIG["gamma"],
        target_update_interval=DQN_CONFIG["target_update_interval"],
        tau=DQN_CONFIG["tau"],
        exploration_fraction=DQN_CONFIG["exploration_fraction"],
        exploration_initial_eps=DQN_CONFIG["exploration_initial_eps"],
        exploration_final_eps=DQN_CONFIG["exploration_final_eps"],
        train_freq=DQN_CONFIG["train_freq"],
        gradient_steps=DQN_CONFIG["gradient_steps"],
        policy_kwargs=dict(net_arch=DQN_CONFIG["net_arch"]),
        tensorboard_log=TENSORBOARD_DIR,
        verbose=DQN_CONFIG["verbose"],
        seed=0,
    )

    print(f"\nQ-Network architecture:\n{model.policy}\n")

    # ── Callbacks ─────────────────────────────────────────────────────────────
    eval_callback = EvalCallback(
        eval_env,
        best_model_save_path=DQN_BEST_MODEL_DIR,
        log_path=os.path.join(MODELS_DIR, "dqn_eval_logs"),
        eval_freq=EVAL_FREQ,   # DQN: single env, so no n_envs division needed
        n_eval_episodes=5,
        deterministic=True,
        render=False,
        verbose=1,
    )
    checkpoint_callback = CheckpointCallback(
        save_freq=CHECKPOINT_FREQ,
        save_path=DQN_CHECKPOINT_DIR,
        name_prefix="dqn_batching",
        verbose=1,
    )
    callbacks = CallbackList([eval_callback, checkpoint_callback])

    # ── Train ─────────────────────────────────────────────────────────────────
    t0 = time.time()
    model.learn(
        total_timesteps=TOTAL_TIMESTEPS,
        callback=callbacks,
        reset_num_timesteps=True,
        tb_log_name="DQN_batching",
        progress_bar=True,
    )
    elapsed = time.time() - t0

    # ── Save final model ──────────────────────────────────────────────────────
    model.save(DQN_FINAL_MODEL_PATH)
    print(f"\n✓  DQN final model saved → {DQN_FINAL_MODEL_PATH}.zip")

    # ── Training summary ──────────────────────────────────────────────────────
    print("\n" + "=" * 60)
    print("  DQN Training Summary")
    print("=" * 60)
    print(f"  Training time     : {elapsed:.1f}s  ({elapsed/60:.1f} min)")
    print(f"  Total timesteps   : {TOTAL_TIMESTEPS:,}")
    print(f"  Throughput        : {TOTAL_TIMESTEPS / elapsed:,.0f} steps/s")
    print(f"  Final model       : {DQN_FINAL_MODEL_PATH}.zip")
    print(f"  Best model        : {DQN_BEST_MODEL_DIR}/best_model.zip")
    print(f"  TensorBoard       : {TENSORBOARD_DIR}")
    print("=" * 60)

    train_env.close()
    eval_env.close()
    return model


if __name__ == "__main__":
    train()
```

class DQNWrapper:
    """Thin wrapper so DQN has the same predict(obs)->int interface as baselines.

    DQN's predict() is the same signature as PPO's, but internally it is doing
    something fundamentally different: it computes Q(s, a) for all actions and
    returns argmax. When deterministic=True, it always picks the greedy action.
    When deterministic=False, it uses ε-greedy during training.

    def __init__(self, model):
        self.model = model

    def predict(self, obs: np.ndarray) -> int:
        action, _ = self.model.predict(obs, deterministic=True)

## 8. What to Expect in Results (and Why)

Based on the properties of this environment, here is what the comparison should look like:

### Expected Ranking
```
PPO ≈ DQN  >  Greedy  >  Random  >>  Cloudflare
```

However the *character* of the two learned policies will be different:

**PPO:**
- Will learn a *stochastic* policy — sometimes serve when queue is at 5 requests, sometimes wait.
- The entropy bonus means it will occasionally explore even after training.
- More stable across different traffic regimes.

**DQN:**
- Will learn a *deterministic* policy — will pick the same action every time it sees the same state.
- May converge faster in very early training (sample efficient from replay buffer).
- Might be more brittle — the Q-values can become overestimated for the `Serve` action, causing it to serve too aggressively ("Q-value overestimation bias"). This is the main known failure mode of DQN.

### Key Metrics to Discuss in Report

| Metric | PPO | DQN | Advantage |
|---|---|---|---|
| Final episode reward | ~600–750 | ~500–700 | PPO |
| Steps to convergence | ~200k | ~150k | DQN |
| Policy stability (std of reward) | Lower | Higher | PPO |
| Training time (wall clock) | ~7 min | ~12 min | PPO |
| Average batch size at serve | ~1–5 | ~1–8 | DQN (bigger batches) |

## 9. Deep Dive: Why DQN Might Struggle Here Specifically

This is good academic content for your report.

### Problem 1: Q-value Overestimation
Standard DQN (as in SB3) uses a technique called **Double DQN under the hood** to combat this, but it still affects the results. When DQN overestimates Q(s, Serve), it will serve too frequently to try to "exploit" a reward it thinks is there but isn't — resulting in smaller batches and lower efficiency rewards.

### Problem 2: Non-Stationarity of the Traffic
The traffic rate changes every episode (random start hour). DQN's Q-values are global — they learn one `Q(s, Serve)` value for a state `s`. But the *true* value of serving in state `s` actually depends on the current traffic rate, which changes. PPO's stochastic policy can represent this uncertainty naturally; DQN's deterministic argmax cannot.

### Problem 3: Sparse Reward Structure
At 10 req/s average, many steps (especially Wait steps) have very small rewards close to 0. DQN can struggle to differentiate between which of two similar states is slightly better, because the Q-value differences are small. PPO with GAE (Generalized Advantage Estimation) is specifically designed to handle this.
