\begin{frame}{The Problem: Every Request Hits the Upstream}
\small
\begin{columns}[T]
\begin{minipage}[t]{0.47\textwidth}
\textbf{Without Batching}
\begin{itemize}
  \small
  \item Each client request triggers one upstream call independently
  \item Under bursty traffic, N simultaneous requests = N upstream calls
  \item Upstream bears full load with no coalescing
  \item Static workarounds (fixed timer, size cap) are blind to traffic state
  \item gzip middleware silently invalidates \texttt{ETag} headers, forcing every conditional request (\texttt{If-None-Match}) to hit upstream with no cache benefit.
\end{itemize}
\end{minipage}
\hfill
\begin{minipage}[t]{0.47\textwidth}
\textbf{With Intelligent Batching}
\begin{itemize}
  \small
  \item Concurrent requests to the same endpoint are parked in a shared slot
  \item One upstream call serves N waiting clients
  \item Flush timing adapts to queue depth, arrival rate, and backend health
  \item The question is not whether to batch - it is \textit{when} to flush
\end{itemize}
\end{minipage}
\end{columns}
\end{frame}

\begin{frame}{Related Work \& Inspiration}
\small
\begin{itemize}
  \item \textbf{1. Cloudflare "Sometimes I Cache" (2024)} \\
  Exponential urgency formula $p(t) = e^{-\lambda \cdot \text{remaining}}$ is optimal for age only decisions. We extend this to a 4 signal observation space conditioning on queue depth and arrival rate.
  
  \vspace{0.6em}
  \item \textbf{2. Cold RL, Bhayani, arXiv:2508.12485 (2025)} \\
  Offline RL for NGINX cache eviction via DQN + ONNX sidecar + 500$\mu$s hard timeout + LRU fallback. Directly validates our offline first training, ONNX serving, and hard deadline fallback pattern.
  
  \vspace{0.6em}
  \item \textbf{3. IEEE 10740859} \\
  Entropy regularisation in PPO prevents policy collapse to always WAIT or always FLUSH. Directly motivated our exploration strategy ($\beta = 0.01$ entropy coefficient).
  
  \vspace{0.6em}
  \item \textbf{4. Sarathi Serve, Agrawal et al., OSDI 2024} \\
  Same throughput and latency tradeoff in LLM inference batching. Different domain (GPU inference vs HTTP proxy), same core tension, validates the problem framing.
\end{itemize}
\end{frame}

\begin{frame}{Our Solution: A Rust Reverse Proxy with RL Flush Control}
\small
\begin{columns}[T]
\begin{column}{0.33\textwidth}
\textbf{Request Flow}
\begin{itemize}
  \item TCP connection accepted via Tokio listener
  \item Semaphore caps max concurrent connections
  \item Router fingerprints request: GET \texttt{->} batch, non-GET \texttt{->} pass-through
\end{itemize}
\end{column}
\begin{column}{0.33\textwidth}
\textbf{Batching Engine}
\begin{itemize}
  \item DashMap registry groups requests by endpoint key
  \item Incoming request parks via \texttt{oneshot} channel
  \item Timer task spawned once per BatchSlot on creation
\end{itemize}
\end{column}
\begin{column}{0.33\textwidth}
\textbf{Flush Decision}
\begin{itemize}
  \item Hard limits checked unconditionally first
  \item RL Agent queried only inside safe envelope
  \item Fan-out: \texttt{mem::take} drains senders, all clients receive response simultaneously
\end{itemize}
\end{column}
\end{columns}

\vspace{1em}
\begin{center}
\textcolor{accent}{Written in Rust - Tokio + Hyper + DashMap. Zero lock contention across concurrent endpoints.}
\end{center}
\end{frame}

\begin{frame}{RL Algorithm: Why PPO with Offline Pre-training?}
\small
\begin{columns}[T]
\begin{column}{0.58\textwidth}
  \textbf{Algorithm Selection}
  
  \vspace{0.5em}
  \begin{tabular}{lp{4.2cm}}
  Not DQN & Experience replay not suited to per-request sub-5ms decision frequency \\
  Not plain PPO & On-policy only; wastes heuristic telemetry from before agent exists \\
  PPO + Offline \checkmark & Warm-start from Kafka replays; entropy bonus $\beta=0.01$ prevents degenerate policy collapse \\
  \end{tabular}
\end{column}
\begin{column}{0.4\textwidth}
  \textbf{Network Architecture}
  \begin{itemize}
    \small\ttfamily
    \item Input: obs [4]
    \item Linear(4 -> 64) + LayerNorm + ReLU
    \item Linear(64 -> 64) + LayerNorm + ReLU
    \item Linear(64 -> 2): WAIT or FLUSH logits
    \item Categorical distribution -> action
  \end{itemize}
\end{column}
\end{columns}

\vspace{1em}
\begin{itemize}
  \item Separate actor and critic networks - no shared layers (non-stationary reward scale).
  \item Trained offline on Kafka \texttt{batch-telemetry}, exported to ONNX, served via gRPC sidecar at $<$1ms inference.
\end{itemize}
\end{frame}
